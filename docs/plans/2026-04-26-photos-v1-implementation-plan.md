# Photos V1 Implementation Plan

Date: 2026-04-26
Status: Proposed, revised after adversarial review and Brooks-Lint plan review on 2026-04-30

## Summary

Photos V1 adds a reusable photo system to BreedWise while keeping the app offline-first and recoverable through backup/restore.

V1 exposes:

- Mare and stallion profile photos.
- Daily log photo attachments.
- Camera capture and device library selection.
- App-private full-size JPEG masters plus generated JPEG thumbnails.
- Full backup/restore of photo metadata and image files.
- A 12-photo limit per daily log.

V1 intentionally does not expose captions, pregnancy check photos, foaling photos, pinch-to-zoom, or saving captured photos back to the device photo library. The schema should allow those later without another redesign.

The adversarial review found two valid architectural holes in the first draft: base64 photos inside a single JSON backup are a memory cliff, and SQLite/photo-file writes need an explicit recovery model. This revision closes those by using a single backup archive file and a files-first persistence strategy with a guarded consistency sweep.

Do not start storage or UI implementation until Phase 0 proves the binary archive path on the current Expo SDK. If that spike fails, stop and revise this plan rather than falling back to base64 or a fully buffered archive.

## Product Decisions

- First exposed surfaces: mare profile photo, stallion profile photo, and daily log attachments.
- Future surfaces: pregnancy checks and foaling records reuse the same attachment model.
- Source options: users can take a photo or choose existing photos from the device library.
- File ownership: BreedWise copies and normalizes selected/captured photos into app-owned storage.
- Device library writes: captured photos are not saved back to the user's photo library in V1.
- Stored quality: keep a full-size app JPEG master plus a smaller JPEG thumbnail. Do not preserve byte-for-byte HEIC/PNG originals in V1.
- Daily log limit: maximum 12 attached photos per daily log.
- Captions: keep nullable caption storage for future use, but do not expose caption UI in V1.
- Backup format: new photo-capable backups become a single `.breedwisebackup` archive containing `backup.json` plus image files. Restore continues to accept existing `.json` backups.

## Architecture

### Dependencies

Install SDK-compatible native dependencies with Expo, not raw `npm install` versions:

```bash
npx expo install expo-image-picker expo-image-manipulator
```

Use the modern `expo-file-system` `File`/`Directory` API for new binary photo/archive work because it supports file objects, bytes, and streams. Keep the existing `expo-file-system/legacy` backup helpers for legacy JSON text files until they are intentionally migrated.

Phase 0 must verify the exact SDK 55 import path and write API before any production storage work:

- Confirm whether the app should import `File`/`Directory` from `expo-file-system` or another SDK-specific path.
- Prove binary writes using `File.write(Uint8Array, { append: true })` or `File.open().writeBytes(...)` against a 5 MB JPEG on Android and iOS.
- Prove the same API can append archive chunks without converting those chunks to base64 strings.

Use a ZIP-compatible archive with a BreedWise-specific `.breedwisebackup` extension. Prefer `fflate` because it is pure JavaScript and avoids adding a native module to the Expo app, but do not add it as a committed dependency until the Phase 0 spike confirms the exact streaming API works in Expo SDK 55. The implementation must use streaming or chunked archive APIs; it must not build one giant base64 string, one fully buffered ZIP `Uint8Array`, or a fully buffered archive string in memory.

Archive backpressure requirements:

- Feed source file bytes to the archive writer in bounded chunks.
- Flush archive output chunks to disk through a sequential async sink.
- Do not read/push the next source chunk until the sink has drained output generated by the current chunk.
- Track pending output bytes in memory and fail the spike if pending archive output grows without bound.
- Measure peak JS heap continuously during archive creation, not only before and after the run.

Update `app.json` permission configuration explicitly. Do not rely on `expo-image-picker` plugin defaults because the default Android manifest includes microphone permission that V1 does not need.

Add `expo-image-picker` as an explicit config plugin entry:

```json
[
  "expo-image-picker",
  {
    "cameraPermission": "BreedWise uses the camera so you can attach horse photos to records.",
    "photosPermission": "BreedWise lets you choose photos from your library to attach to horse records.",
    "microphonePermission": false
  }
]
```

Keep the existing `app.config.js` merge behavior in mind: plugin entries added to `app.json` flow through that config file.

Do not request photo-library write permission in V1.

### Data Model

Add two tables in the next available migration after rebasing onto `main`. Based on the current repo this is expected to be `028_photos`, but do not hard-code that name until the branch is rebased. Migrations are centralized and other feature branches may consume the next ID first.

Migration requirements:

- Fresh install and upgraded install must produce the same table, constraint, and index shape.
- Add `shouldSkip` predicates using `tableExists` for both tables and `indexExists` for all new indexes, including the partial profile index.
- If the migration needs repair helpers for already-partial state, add them in the migration file rather than relying on manual cleanup.
- Migration tests must cover fresh schema, upgraded schema, and skip/repair behavior.

`photo_assets` stores file metadata:

- `id TEXT PRIMARY KEY`
- `master_relative_path TEXT NOT NULL`
- `thumbnail_relative_path TEXT NOT NULL`
- `master_mime_type TEXT NOT NULL CHECK (master_mime_type = 'image/jpeg')`
- `thumbnail_mime_type TEXT NOT NULL CHECK (thumbnail_mime_type = 'image/jpeg')`
- `width INTEGER NOT NULL CHECK (width > 0)`
- `height INTEGER NOT NULL CHECK (height > 0)`
- `file_size_bytes INTEGER NOT NULL CHECK (file_size_bytes > 0)`
- `source_kind TEXT NOT NULL CHECK (source_kind IN ('camera', 'library', 'imported'))`
- `created_at TEXT NOT NULL` storing ISO 8601 timestamps
- `updated_at TEXT NOT NULL` storing ISO 8601 timestamps

`photo_attachments` links assets to domain records:

- `id TEXT PRIMARY KEY`
- `photo_asset_id TEXT NOT NULL`
- `owner_type TEXT NOT NULL CHECK (owner_type IN ('mare', 'stallion', 'dailyLog', 'pregnancyCheck', 'foalingRecord'))`
- `owner_id TEXT NOT NULL`
- `role TEXT NOT NULL CHECK (role IN ('profile', 'attachment'))`
- `sort_order INTEGER NOT NULL DEFAULT 0 CHECK (sort_order >= 0)`
- `caption TEXT CHECK (caption IS NULL OR length(caption) <= 500)`
- `created_at TEXT NOT NULL` storing ISO 8601 timestamps
- `updated_at TEXT NOT NULL` storing ISO 8601 timestamps
- FK `photo_asset_id` references `photo_assets(id)` with `ON UPDATE CASCADE ON DELETE RESTRICT`

The `RESTRICT` delete is intentional: assets should only be removed after attachments are gone, so mistaken early deletes surface as FK errors.

The polymorphic owner model is an explicit tradeoff. SQLite cannot enforce `owner_id` across multiple owner tables, so the CHECK constraint, repository cleanup, restore validation, and consistency sweep must stay in sync. Keep valid owner values in `src/models/enums.ts` and reuse those constants in repository code, validators, tests, and the sweep. Migration SQL still freezes the owner values for that schema version, so adding a future owner type requires a new migration and backup schema update.

Indexes and constraints:

- Index attachments by `(owner_type, owner_id, role, sort_order, created_at, id)`.
- Enforce one profile attachment per owner with a unique partial index where `role = 'profile'`.
- Do not make `sort_order` unique; reorder operations must renumber from zero, and list queries must tie-break by `sort_order`, `created_at`, then `id`.
- Enforce the 12-photo daily log attachment limit in repository/service logic.

Add domain definitions:

- `src/models/enums.ts`: `PhotoOwnerType`, `PhotoAttachmentRole`, `PhotoSourceKind`, plus arrays/constants used by validators.
- `src/models/types.ts`: `PhotoAsset`, `PhotoAttachment`, repository input types, and resolved display DTOs.
- `src/utils/outcomeDisplay.ts`: `formatPhotoOwnerType` before any owner type is rendered in user-facing copy.

Do not add photo columns to `mares`, `stallions`, or `daily_logs`.

### File Storage

Store photo files under app-owned document storage:

```text
FileSystem.documentDirectory/
  photo-assets/
    <storageId>/
      master.jpg
      thumbnail.jpg
  photo-drafts/
    <draftId>/
      master.jpg
      thumbnail.jpg
```

Persist only relative paths in SQLite. Resolve them at runtime against `FileSystem.documentDirectory`.

Import normalization:

- Convert every selected/captured source to a JPEG master using `expo-image-manipulator`.
- Normalize EXIF orientation during conversion.
- Set a maximum master long edge of 2400 px.
- Start around JPEG quality `0.85`; retry lower quality or smaller dimensions if the master exceeds 2 MB.
- Reject the import with a clear message if it cannot be normalized under the file-size cap.
- If image manipulation throws because the source is corrupt, unsupported, out of memory, or otherwise unreadable, keep existing staged photos unchanged and show: `This photo couldn't be processed. Try a different photo.`
- Generate a thumbnail with longest edge around 512 px.

This means "master" is the full-size BreedWise copy, not a byte-for-byte source original. That tradeoff is deliberate to avoid HEIC display failures, huge backups, and inconsistent EXIF rotation across platforms.

### SQLite and Filesystem Consistency

SQLite and the document directory cannot be updated atomically together. Use this recovery model everywhere:

1. Write or finalize files first.
2. Insert/update SQLite metadata in a transaction second.
3. After commit, delete old unreferenced files as best effort.
4. On app bootstrap and before backup/restore, run a photo consistency sweep through the photo-file write queue.

Add `src/storage/photoFiles/writeQueue.ts`:

- Export `runExclusivePhotoFileWork<T>(work: () => Promise<T>): Promise<T>`.
- All draft finalization, asset deletion, archive write/read file extraction, and consistency sweep code must run through this queue.
- Repository-only SQLite work does not use this queue unless it also finalizes or deletes files.
- The queue protects in-process races. The sweep must also ignore finalized `photo-assets/` directories whose newest file timestamp is less than 60 seconds old so a restarted app cannot delete files from a just-resumed write path before metadata appears.

Bootstrap and restore ordering:

- `useAppBootstrap` should not set `isReady` until `initDb()` and `runPhotoConsistencySweep()` have both completed.
- Backup creation must acquire the queue, run the sweep, then serialize metadata and archive files.
- Restore must acquire the queue for the whole file-write plus DB-transaction sequence.

The consistency sweep must:

- Delete `photo-drafts/` directories older than 24 hours.
- Delete `photo-assets/` directories that have no `photo_assets` row and are outside the 60-second grace window.
- Delete attachment and asset metadata when the DB row points to missing `master.jpg` or `thumbnail.jpg`; delete `photo_attachments` rows first, then `photo_assets` rows, matching backup delete order.
- Delete `photo_attachments` rows whose polymorphic owner no longer exists, then delete any newly orphaned `photo_assets` metadata/files.
- Never delete files that are still referenced by valid DB rows.

This bounds every crash case:

- If file finalization succeeds but DB insert fails, the next sweep removes orphan files after the grace window.
- If DB commit succeeds but old file cleanup fails, the next sweep removes old unreferenced files.
- If restore writes new files but DB restore rolls back, the next sweep removes unreferenced restore files.
- If a user or OS removes image files but DB rows remain, the next sweep removes broken metadata and UI falls back cleanly.
- If a hard-deleted owner row is removed before photo cleanup completes, the next sweep removes dangling attachment metadata and any resulting orphan assets.

### Repositories and Hooks

Add `src/storage/repositories/photos.ts` and export it from the repository index.

Repository responsibilities:

- List profile photo for an owner.
- List attachment photos for an owner.
- Set or replace a profile photo.
- Clear a profile photo.
- Add daily log attachment photos.
- Replace daily log attachment ordering.
- Delete a single attachment.
- Bulk delete attachments for hard-deleted owners. V1 must wire this for daily logs, pregnancy checks, and foaling records because all three can be physically deleted while the photo backup schema can preserve those owner types.
- Delete orphaned `photo_assets` metadata after the last attachment is removed.
- Enforce the 12-photo daily log attachment limit.

Profile replace flow:

- Finalize the new asset files first.
- Inside one DB transaction, update the existing profile attachment in place when it exists; otherwise insert it.
- After commit, delete the old asset if no attachment references it.
- Add a repository test that replaces the same profile photo 50 times and asserts there is one profile attachment and no referenced-orphan metadata.

Soft delete policy:

- Soft-deleting mares or stallions keeps profile photos.
- Do not detach profile photos during soft delete.
- Bulk owner cleanup is used for hard deletes, including daily log, pregnancy check, and foaling record delete.
- Convert pregnancy check and foaling record delete flows to use transactions when photo cleanup is wired, because they currently run a single `DELETE` and will need same-transaction attachment removal.
- If mare or stallion hard-delete behavior is added later, it must first delete their photo attachments inside the same owner-delete transaction.

Daily log save policy:

- Add a photo-aware save orchestration path rather than nesting transactions inside the current daily log save.
- For create/edit, finalize new files first, then run one SQLite transaction that saves the daily log and photo metadata together.
- If photo persistence fails, keep the user on the review step with staged photos intact and show a retryable error. Do not navigate away as if photos saved.
- If the daily log save fails after file finalization, the consistency sweep removes orphaned finalized files.

Add hooks under `src/hooks/`:

- `useProfilePhotoDraft` for mare/stallion form staging.
- `usePhotoDrafts` for daily log staged attachment add/remove/reorder.

Screens should call hooks, not repositories or file APIs directly.

## Backup and Restore

Bump backup schema from the current schema to the next available schema after rebasing. Based on the current repo, this is expected to be v12 because v10 is breeding-record timestamps and v11 is dashboard tasks.

New backup artifact:

```text
breedwise-backup-v12-YYYYMMDD-HHmmss.breedwisebackup
  backup.json
  photo-assets/
    <photoAssetId>/
      master.jpg
      thumbnail.jpg
```

Safety snapshots use the same single-file archive model:

```text
breedwise-safety-backup-v12-YYYYMMDD-HHmmss.breedwisebackup
```

The `.breedwisebackup` file is a ZIP-compatible archive with a custom extension. `backup.json` remains human-readable inside the archive.

Restore compatibility:

- Continue accepting existing `.json` backups for schema v1-v11.
- Accept `.breedwisebackup` archives for the photo backup schema and later, expected v12+.
- Reject `.breedwisebackup` files with a schema greater than `BACKUP_SCHEMA_VERSION_CURRENT` using the existing newer-app message.
- The document picker must allow both JSON and BreedWise backup archive files.

File I/O compatibility tasks:

- Update `src/storage/backup/fileIO.ts` filename prefixes. The current `breedwise-backup-v5-` and `breedwise-safety-backup-v5-` prefixes are cosmetic but misleading; Phase 3 should either derive the prefix from `BACKUP_SCHEMA_VERSION_CURRENT` or centralize the current backup filename version so it cannot drift silently again.
- Manual and safety snapshot filenames use `.breedwisebackup` for the photo schema while legacy JSON restore remains supported.
- Parameterize `shareFileIfAvailable` MIME handling: JSON backups use `application/json`; BreedWise archives use `application/zip` or a documented custom archive MIME if platform testing proves it works better.
- Update `pickBackupFile` to allow JSON and archive selection. Default to `*/*` plus explicit extension validation for `.json` and `.breedwisebackup`; only narrow MIME filters if Android and iOS testing proves both extensions remain selectable.
- Update tests/mocks so a selected `.breedwisebackup` file is not filtered out.
- Update `listSafetySnapshots` and pruning to understand both legacy `.json` snapshots and archive snapshots during the transition. For `.breedwisebackup` files, listing must open the archive, extract `backup.json`, parse/validate that metadata, and then close the archive without extracting photo files. Pruning still keeps the latest 3 valid snapshots by parsed creation timestamp, regardless of extension.

Backup `backup.json` contains raw DB rows and photo file manifest entries, but not base64 image data. Photo files live as binary archive entries.

Add managed tables:

- `photo_assets`
- `photo_attachments`

Phase 3 must update all backup plumbing explicitly:

- `BACKUP_SCHEMA_VERSION_V12` or the next available equivalent
- `BACKUP_SCHEMA_VERSION_CURRENT`
- `BACKUP_TABLE_NAMES`
- `BACKUP_DELETE_ORDER`
- `BACKUP_INSERT_ORDER`
- raw row types
- `BackupTablesV12 = BackupTablesV11 & { photo_assets: ...; photo_attachments: ... }` or the next available equivalent
- `BackupEnvelopeV12` or the next available equivalent
- serializer queries
- validator table requirements
- restore normalization and insert/delete order
- safety snapshot listing and pruning
- backup fixtures and tests

Backup order requirements:

- Add `photo_assets` and `photo_attachments` to managed tables.
- Insert `photo_assets` before `photo_attachments`.
- Delete `photo_attachments` before `photo_assets`.
- Insert `photo_attachments` only after the owner rows it references have been restored.
- The consistency sweep must use the same attachment-before-asset metadata delete order.
- Add a binary fixture in `src/utils/backup/testFixtures.ts` or a backup test helper: a tiny checked-in JPEG byte literal is enough to prove the archive path is not metadata-only.

Restore behavior:

- Validate every photo-schema `photo_asset` has both archive file entries.
- Validate every attachment references an existing photo asset.
- Validate `mare`, `stallion`, `dailyLog`, `pregnancyCheck`, and `foalingRecord` owner references.
- Accept attachments for soft-deleted mares/stallions if the owner row exists; backups preserve hidden data.
- Accept `pregnancyCheck` and `foalingRecord` attachments even though V1 UI does not expose them; preserve them invisibly for forward compatibility.
- Because `pregnancyCheck` and `foalingRecord` rows are currently hard-deletable, their delete repositories and the consistency sweep must remove dangling photo attachments even though the UI does not expose those photo surfaces yet.
- Reject unknown owner types in the photo backup schema. Future owner types require a new backup schema version.
- During restore, write photo files first to collision-free relative paths, then run the SQLite full-replace transaction with those rewritten paths.
- Do not delete old photo files before the DB transaction commits; old files become unreferenced and are removed by the consistency sweep.
- Emit full data invalidation after restore.

The old base64-in-JSON approach is rejected for V1 because realistic photo counts can OOM the JS runtime and create unusably large JSON files. A fully buffered ZIP is also rejected for the same reason.

## Review Resolution Notes

Valid adversarial review points incorporated here:

- JSON base64 backup memory risk: replaced with a single archive artifact.
- Fully buffered ZIP memory risk: Phase 0 must prove streaming/chunked archive writes with a heap ceiling.
- SQLite/filesystem atomicity: files-first writes plus queued consistency sweep, bootstrap gating, and a 60-second grace window for newly finalized files.
- Profile replacement unique-index risk: update-in-place transaction and repeated-replace test.
- Soft-delete ambiguity: keep mare/stallion profile photos on soft delete.
- Polymorphic owner risk: owner type CHECK plus explicit restore validation.
- Future owner dangling risk: hard-delete cleanup and consistency sweep coverage for daily logs, pregnancy checks, and foaling records.
- Draft leaks: boot-time and pre-backup stale draft cleanup.
- Literal feature flag risk: testable non-literal boolean helper.
- HEIC/orientation risk: normalize all app masters and thumbnails to JPEG.
- Multi-select cap and permission behavior: explicit UI rules.
- Picker ordering risk: use platform-supported `selectionLimit`/ordered selection but only rely on returned-array order.
- Unneeded microphone permission risk: explicit image-picker config sets `microphonePermission: false`.
- Daily log save transaction risk: photo-aware save orchestration.
- Migration ID risk: next available ID after rebase, expected `028_photos` only if current main stays unchanged.
- Backup schema churn: current repo baseline is v11, photos are expected to use v12, and Phase 3 lists explicit file I/O and backup type changes.
- Native API testability risk: Jest mocks for screen tests plus Vitest mocks or injected adapters for unit tests.
- Safety snapshot scale: archive snapshots, not base64 JSON snapshots.
- Safety snapshot compatibility: listing and pruning must handle both legacy `.json` snapshots and `.breedwisebackup` snapshots.
- Header layout and thumbnail overflow: fixed avatar and horizontal strips.
- Viewer expectations: close/loading/error/swipe in V1; pinch-to-zoom deferred.
